# ADR 0001：内容存储边界

- 状态：Accepted
- 日期：2026-07-17

## 背景

作品集由单一管理员维护，站点配置、项目和博客内容当前作为一个受 Pydantic 严格校验的 JSON 文档保存。每次修改使用 ETag 与 PostgreSQL advisory lock 防止静默覆盖，并在写入前保留完整修订快照。

这种模型能让小型单人站点保持简单，但整站读取、完整快照和媒体引用递归扫描的成本会随文章、修订和媒体数量共同增长。

## 决策

当前继续将站点配置和内容保存在单个文档中，不在没有性能证据时提前引入多表 CMS。以下约束是该决策的一部分：

- 内容大小和修订数量必须有硬上限。
- 所有写入必须通过完整 schema 校验、显式发布状态和并发版本条件。
- 公共读取应支持 ETag，并只返回已发布且到达发布时间的内容。
- 媒体列表与清理不得无限制加载全部修订和全部文件。
- 备份恢复必须同时覆盖数据库文档、修订和上传媒体。

## 重新评估条件

出现任一条件时，应基于基准数据评估把文章、媒体引用和修订拆为独立表：

- 引入多个作者、审核流程或字段级权限。
- 文章接近 100 篇，或媒体接近 1000 个。
- 单个内容文档接近配置上限。
- 媒体列表、内容保存或修订恢复超过既定延迟或内存预算。
- 需要数据库级搜索、分页、标签统计或按文章独立发布。

迁移时站点级展示配置可以继续保留 JSON；文章、媒体引用和修订应拥有稳定主键、外键及可回滚的 Alembic 数据迁移。

## 影响

短期内代码和备份流程保持简单，但必须持续进行边界测试和性能观测。该决策不允许以“个人项目”为理由移除并发控制、发布状态、大小限制或恢复验证。
